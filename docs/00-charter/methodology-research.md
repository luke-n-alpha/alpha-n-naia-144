Phase 2: 투표 — 발견사항 매칭 → CONFIRMED/CONTESTED/AUTO-DISMISSED

+ R=2 가드 / 전원 플래그 가드
+ JSONL 이벤트 로깅

## 4. 설계 과제 (연구 중)

### 4.1 검증→생성 확장

특허는 "이미 존재하는 산출물을 검증"하는 구조. 게임 개발은 "산출물을 생성"해야 함.

- **옵션 A**: 생성은 단일 에이전트, 검증만 특허 메커니즘 적용
- **옵션 B**: 동일 작업을 N개 에이전트가 독립 생성 → 보팅으로 최적안 선택 → 검증도 특허 메커니즘
- **옵션 C**: 혼합 — 핵심 결정(아키텍처, 타일셋)은 보팅, 구현은 단일 + 검증

### 4.2 조직 구성

에이전트 도메인을 게임 개발에 맞게 재정의 필요. 후보:
- 엔진(터미널/폰트/렌더), 비주얼(타일셋/자모 매핑), 게임로직(월드/NPC/룰), 시나리오(한국어 텍스트), 사이즈(1.44MB 예산), 품질(검증)

### 4.3 오케스트레이터

- 현재 세션(opencode/GLM)이 오케스트레이터 역할
- Gemini, Codex를 리뷰/생성 에이전트로 활용

### 5.1 질문

Gemini CLI와 Codex CLI에 동일 질문을 전달하여 독립 의견 수집:

1. 이 특허 방법론을 게임 개발(생성+검증)에 응용할 때 가장 큰 기술적 리스크는?
2. "검증" 중심 설계를 "생성"에까지 확장하려면 어떤 보완이 필요한가?
3. 3개 AI(Gemini, Codex, GLM)만으로 이 방법론을 의미 있게 실행할 수 있는가? 최소 에이전트 수 제안?
4. review-pass v3이 특허의 핵심 메커니즘을 구현하지 않은 상태에서, 이 프로젝트에서 우선 구현해야 할 메커니즘 Top 3는?

### 5.2 응답

#### Codex (GPT-5.4) — 2026-05-22

**Q1. 가장 큰 기술적 리스크**

> 생성 단계에서 독립성은 높이고 시스템 일관성은 잃는 것. 검증은 같은 산출물을 두고 다수 판단을 모을 수 있지만, 생성은 에이전트마다 산출물 자체가 달라진다. 보팅이 "정답 판정"이 아니라 "서로 다른 설계안 비교"가 되고, 책임 경계가 흐려진다. 특히 1.44MB, 자체 터미널/폰트 렌더, 한글 자모 타일, 게임성이 강하게 결합돼 있어서, 에이전트별 로컬 최적화가 전체 예산과 엔진 제약을 깨뜨릴 가능성이 높다. 최대 리스크는 환각 그 자체보다 **다중 생성이 아키텍처 일관성과 용량 예산을 파괴하는 것**.

**Q2. 생성 확장 보완**

> `판정 규칙`이 아니라 `생성 계약`이 먼저 필요. 세 가지 보완:
> 1. **공통 스펙 강제**: 메모리/디스켓 용량 예산, 렌더링 API, 타일 규격, 텍스트 인코딩, 자모 조합 규칙 같은 하드 제약
> 2. **비교 가능한 단위로 쪼개기**: 전체 게임을 통째로 여러 AI가 만드는 게 아니라, 폰트 렌더러, 타일셋 매핑, 월드 규칙, 시나리오, 압축/사이즈 최적화처럼 좁은 산출물 단위로 독립 생성
> 3. **Merge Arbiter**: 단순 보팅이 아니라 여러 안에서 채택 가능한 부분만 조합하고, 충돌은 상위 판단 모델이 해소
>
> 생성 확장판 구조: `독립 생성 → 제약 검사 → 후보 병합 → 다중 검증`

**Q3. 최소 에이전트 수**

> 3개 AI로 의미 있는 실행 가능. 단 축약형 방법론으로 운영.
> - **현실 최소**: 3에이전트 (생성 1, 검증 1, 중재 1) — 하지만 다수 독립 보팅이 약화
> - **실용적 하한**: 5에이전트 (생성 2, 검증 2, 중재 1)
> - **안정**: 7에이전트
> - 동일 모델을 서로 다른 컨텍스트로 분리해 "가상 에이전트" 운용은 가능하나, 역할 분리를 더 엄격히 해야 함

**Q4. 우선 구현 Top 3**

1. **안티앵커링 교체**: 생성/검증 모두에서 가장 먼저 필요. 이전 판단이 다음 에이전트를 오염시키면 다중화 의미가 사라짐
2. **7-신호 건강 점수 + 전원 플래그 가드**: "누가 왜 부실했는지"를 계량하는 장치. 에이전트 품질을 수치화하지 않으면 보팅도 노이즈에 끌림
3. **스트라이크 관리 + 동적 퇴출**: 반복적으로 헛판단하거나 문맥을 못 읽는 에이전트를 계속 투입하면 전체 비용만 늘고 품질은 안 오름

**한 줄 요약**: 지금은 검증기 숫자보다 독립성 유지, 생성 계약, 저품질 에이전트 통제를 먼저 구현해야 함. 생성까지 포함하려면 "다수 보팅"보다 "제약된 역할 분해"가 먼저.

#### Gemini — 2026-05-22

**Q1. 가장 큰 기술적 리스크**

> N개 에이전트가 독립적으로 생성할 경우, 각 파트(엔진, 로직, 비주얼) 간의 인터페이스나 데이터 규격이 파편화되어 하나의 게임으로 정합성 있게 통합되지 못할 위험이 가장 큼.

**Q2. 생성 확장 보완**

> - N개 독립 생성물 중 최적안을 고르는 **'생성물 평가 보팅 메커니즘'** 필요
> - 생성 모델, 검증 모델, 중재 모델로 **역할 명확 분리** 및 **모델 계층화** 필요

**Q3. 최소 에이전트 수**

> 실행 가능. Gemini(검증), Codex(생성), GLM(중재/오케스트레이터) 역할 분담.
> 투표 시 동점 방지 및 교차 검증 객관성 확보를 위해 **최소 R=3** 권장 (R=2는 기능 제한적 가드 상태).

**Q4. 우선 구현 Top 3**

1. **스트라이크 관리**: 저품질 응답 반복 에이전트에 대한 적대적 품질 제어
2. **안티앵커링**: 생성물의 자신감이나 타 리뷰어 의견에 휘둘리지 않는 독립적 검증 보장
3. **컨텍스트 드리프트 감지**: 생성 의도와 검증 기준 사이의 괴리를 실시간 추적하여 정렬 유지
