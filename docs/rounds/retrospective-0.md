## 2. 무엇을 놓쳤는가 (드리프트)

### 2.1 게임 기획이 없다

계획한 모듈 순서:
```
A1. 생성 계약 ✅
A2. 타일셋 (GE → CX 검증)
A3. 폰트 렌더러 (CX → GE 검증)
```

그런데 **무슨 게임인지 아직 정의 안 했다.** cafelua + naia + Alpha라는 세계관은 있지만:
- 플레이어가 무엇을 하는가?
- 챕터 구조는?
- 승리/패배 조건은?
- 화면은 어떻게 전환되는가?
- 턴제인가, 실시간인가?

이게 없이 엔진을 먼저 만들면, 엔진이 게임에 맞지 않을 수 있다. 인터페이스 계약에 `SceneType`이 이미 정의되어 있지만(TITLE, CAFE, ATELIER, BATTLE, DIALOG, ENDING), 이건 OC가 임의로 넣은 것이지 CX/GE 합의를 거친 게임 디자인이 아니다.

### 2.2 특허 메커니즘이 아직 실동 안 함

방법론 설계는 했지만, 실제 코드 생성 라운드에서:
- 스트라이크 관리, 건강 점수, 퇴출 메커니즘이 한 번도 실행 안 됨
- 안티앵커링이 중요하다고 합의했지만, 실제로 독립 생성을 시도한 적이 없음
- 이벤트 로그(JSONL) 포맷만 정의되고 한 줄도 기록 안 됨

**실험의 본질(특허 방법론 실증)이 아직 시작 안 됨.**

### 2.3 메타 작업 비율이 너무 높음

전체 산출물이 문서만. 물론 Phase 0-1은 설계 단계니까 문서가 주된 산출물인 게 맞지만, "활동 자체가 보고서" 원칙상 실제 코드 생성 라운드에서 무슨 일이 일어나는지가 핵심 관찰 대상이다. 아직 그 관찰을 시작도 안 함.

## 3. 중심 — 이 실험의 진짜 목적

### 3.1 게임이 아니다

이 프로젝트는 "게임을 만드는 것"이 1차 목적이 아니다. 게임은 **실험 매개체**다.

**진짜 목적:**

> **SDLC 기반에서 AI 에이전트들이 체계적으로 소프트웨어를 생산하기 위한 하네스 아키텍처를 설계·실증하고, 그 결과를 naia-adk에 이식하는 것.**

구체적으로:
1. 특허 10-2026-0056403(멀티에이전트 상호 검증)의 메커니즘을 **생성 단계까지 확장**하는 방법론을 실험
2. 생성 계약, 독립 생성, 다중 검증, 수렴 루프가 **실제 코드 산출**에서 어떻게 작동하는지 관찰
3. 관찰 결과를 **재사용 가능한 하네스 아키텍처**로 일반화
4. 완성된 하네스를 **naia-adk에 이식** — naia-adk의 모든 프로젝트에서 AI 에이전트가 SDLC에 따라 체계적으로 소프트웨어를 생산할 수 있도록

### 3.2 두 가지 산출물

| 산출물 | 성격 | 대상 |
|--------|------|------|
| **1.44MB 디스켓 게임** | 실증 매개체. 공모전 출품. | 외부 (공개) |
| **AI Agent Harness Architecture** | 방법론 실증 결과. 재사용 가능한 프로토콜·도구·템플릿. | naia-adk 이식 |

### 3.3 사람들이 볼 수 있어야 한다

이 프로젝트는 오픈소스로 공개된다. 읽는 사람(개발자, 연구자, 공모전 심사위원)이 이해할 수 있어야:
- 무엇을 실험했는지
- 어떤 구조로 AI 에이전트가 협업했는지
- 무슨 일이 일어났는지 (성공도 실패도)
- 결과를 어떻게 재사용할 수 있는지

따라서 문서는 실험 노트이자 동시에 **읽을 수 있는 보고서**여야 한다.

### 3.4 핵심 질문 (수정)

기존 질문(methodology-research.md §1.2)에 상위 질문을 추가:

> 2. 다수 에이전트 보팅 + 적대적 크로스리뷰가 실제 프로젝트에서 어떻게 작동하는가?
> 3. 결과를 naia-adk에 어떻게 이식하는가?

Phase D: 통합·제출

## 5. 교훈

1. **설계에서 실행으로 넘어가는 경계가 명확해야 함.** 문서가 충분해 보여도 "실제로 돌아가는가?"는 직접 해봐야 안다.
2. **순서 의존성을 과소평가함.** 엔진이 게임에 의존하는데 게임 기획을 Phase C로 미룬 건 설계 오류.
3. **회고가 없으면 드리프트를 모름.** 이 회고 자체가 없었으면 A-2(타일셋)부터 시작했을 것이고, 게임 기획 누락을 나중에야 발견했을 것.
4. **목적과 수단을 혼동함.** 게임(수단)이 아니라 하네스(목적)에 집중해야 하는데, 모듈 순서가 게임 중심으로 짜여 있었다. 하네스 관점에서는 "각 라운드에서 무엇을 관찰해야 하는가"가 우선.
