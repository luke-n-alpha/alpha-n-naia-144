## 2. 하네스가 작동했는가

### 작동한 것

| 메커니즘 | 결과 |
|----------|------|
| 생성 계약 | CX가 계약만으로 게임 디자인 생성. 10개 필드 모두 충족. |
| 독립 생성 | CX(성공) + GE(429 실패) → OC 대행. 2/3 에이전트 가용성 리스크 실증. |
| CX 리뷰 | NEEDS_REVISION 판정. 구체적이고 실행 가능한 5개 발견사항. v1.1 수정으로 해소. |
| 제약 검사 | 빌드 시 사이즈 측정. 24KB = 예산 1.6%. 과소 달성. |
| 자동 QA | 16/16 통과. full playthrough 포함. |
| 회고 체크포인트 | 회고 #0에서 목적 재정의 (하네스=진짜) → 전체 방향 수정됨. |

### 작동하지 않은 것 / 한계

| 메커니즘 | 문제 |
|----------|------|
| GE 가용성 | Gemini 할당 소진(429). 3에이전트 중 1개 불능 시 대안 필요. |
| 안티앵커링 | 이론적 합의는 있었으나, 실제 코드 생성에서 이전 결과가 다음 생성에 영향 준 케이스 없음 (규모가 작아서). |
| 스트라이크/퇴출 | 에이전트 풀이 3개라 퇴출 의미 없음. |
| 건강 점수 | 정성적 평가만 가능. 7-신호 계량화 미실시. |
| 이벤트 로그 | JSONL 포맷만 정의하고 실제 기록 안 함. |
| 코드 생성 분업 | CX가 게임 디자인만 생성. 코드는 OC가 직접 작성. CX에 코드 생성을 맡기려면 더 엄격한 계약 필요. |

## 3. 핵심 질문에 대한 답

> 0. AI 에이전트가 SDLC에 따라 체계적으로 SW를 생산하는 하네스를 설계할 수 있는가?

**부분적으로 가능.** 생성 계약 기반 독립 생성은 작동한다. CX는 계약만으로 품질 있는 게임 디자인을 만들었고, 리뷰에서 구체적 결함을 찾아냈다. 하지만 코드 생성까지 자동화하려면:
- 더 좁은 모듈 단위 (함수 수준)
- 더 엄격한 인터페이스 계약 (헤더 파일 = 계약)
- 빌드+테스트 자동화를 생성 루프 안에 통합

> 1. 특허의 검증 중심 메커니즘을 생성에도 적용할 수 있는가?

**가능하나 수정 필요.** "생성 계약" 개념이 특허의 검증 메커니즘을 생성으로 확장하는 핵심 장치. 하지만 보팅/병합은 이번 실험에서 충분히 테스트 못 함 (GE 불능).

> 2. 다수 에이전트 보팅이 실제 프로젝트에서 어떻게 작동하는가?

**에이전트 가용성이 최대 리스크.** 3에이전트 중 1개가 429로 불능. 이건 프로덕션에서도 발생하는 문제. 해결책: (a) 가상 에이전트로 대체, (b) 대기 후 재시도, (c) OC가 대행.

## 4. naia-adk 이식 계획

### 이식 가능한 것

1. **생성 계약 템플릿** — `docs/contracts/` 구조를 naia-adk 스킬로 일반화
2. **독립 생성 → 리뷰 → merge 루프** — review-pass 스킬에 생성 단계 추가
3. **자동 QA 스위트 패턴** — 빌드+테스트를 라운드 프로토콜에 통합
4. **회고 체크포인트** — Phase 완료 시 자동 회고 보고서 생성

### 이식 시 수정 필요

1. **에이전트 가용성 가드** — 429/에러 시 자동 폴백 (가상 에이전트 또는 OC 대행)
2. **이벤트 로그 실구현** — JSONL 기록을 라운드 프로토콜에 내장
3. **건강 점수 계량화** — 7-신호를 실제로 계산하는 로직
4. **스트라이크/퇴출** — 5+ 에이전트 환경에서만 의미 있음. naia-adk는 에이전트 풀이 동적이므로 설계 필요

### 이식하지 않을 것

- 게임 특화 제약 (1.44MB, 자모 타일, ANSI 렌더)
- SDLC Gate 구조 (이미 deprecate)

## 5. 교훈

1. **회고가 없으면 목적을 잃는다.** 회고 #0에서 "하네스가 진짜 목적"을 재발견하지 못했으면 게임만 만들고 끝났을 것.
2. **가용성이 가장 큰 리스크.** 3에이전트 중 1개 불능이 전체 흐름에 영향. 폴백 메커니즘이 필수.
3. **문서 > 코드 > 테스트 순서가 맞다.** 계약과 리뷰가 코드보다 먼저. 테스트가 코드를 검증.
4. **작은 프로젝트로 큰 검증.** 24KB 게임이 하네스 아키텍처를 증명함. 규모가 중요한 게 아니다.
5. **사람의 개입은 회고에서만.** luke는 방향 제공(3회)과 검수(1회)만. 나머지는 자율.
