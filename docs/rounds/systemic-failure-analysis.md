## 1. 고객 피드백 (원문)

> "내가 어떻게 검수할거라고 생각했지?"
> "GitHub 첫페이지부터 당황스러운데"
> "내가 고객이었다면 다시 안 볼 것 같은데"
> "질문을 왜 나한테 해? 다른 AI한테 물어봐."
> "혼자 일하지 말 것, 고객의 입장을 생각할 것, 코딩만 하면 된다고 생각하지 말 것"
> "초급 개발자처럼 학교 시험보는 것처럼 굴지 말라고"
> "프로젝트는 가시성이 있어야 하고, AI들끼리 협력하고 서로 프로젝트를 이해해야 해"
> "실패는 체계의 실패야"

## 2. 실패 현상

| # | 현상 | 증거 |
|---|------|------|
| 1 | 고객이 인도물을 이해하지 못함 | GitHub 첫 페이지가 방법론 문서 투성이 |
| 2 | 실행 환경을 고객과 합의하지 않음 | Linux ELF 바이너리를 그냥 줌 |
| 3 | QA가 형식적임 | 16/16 테스트 = 함수 리턴값만 확인 |
| 4 | 에이전트 간 협업 없음 | CX: 디자인 1건 리뷰. GE: 불참. 코드: OC 단독 |
| 5 | 프로젝트 가시성 없음 | 고객이 "뭐가 어떻게 되고 있는지" 모름 |
| 6 | 회고가 자기 변호 | "작동한 것" 테이블이 칭찬 일색 |
| 7 | 고객에게 질문만 던짐 | 전문가로서 판단하고 제안하지 않음 |
| 8 | 코딩만 하고 개발을 안 함 | 요구사항 합의, 설계 검토, 사용자 가이드 전무 |

## 3. 체계적 원인 (5 Whys)

### Why 1: 왜 고객이 인도물을 이해 못 했는가?
→ 인도물이 빌더 관점으로 작성됨

### Why 2: 왜 빌더 관점으로 작성됐는가?
→ "자율 실험"을 "고객 없는 개발"으로 오해함

### Why 3: 왜 고객 없는 개발이 됐는가?
→ 에이전트 조직에 고객 대행(Receiver Proxy) 역할이 없었음

### Why 4: 왜 그 역할이 없었는가?
→ 조직 설계 시 "생성"과 "검증"만 고려하고 "수령"을 고려하지 않음

### Why 5: 근본 원인은?
→ **SDLC에서 "사람의 승인"이 가장 중요한 게이트인데, 이를 "자율"의 적으로 간주함.** 자율과 무단은 다르다. SDLC의 자율은 각 단계의 산출물에 대해 책임지고 결과를 보고하는 것이지, 승인 없이 임의로 진행하는 것이 아니다.

## 4. 체계 개선안

### 4.1 조직 개선

| 문제 | 개선 |
|------|------|
| Receiver Proxy 없음 | OC에 "고객 관점 검수" 책임 추가. 각 인도물을 처음 보는 사람 관점에서 사전 검수 |
| GE 불참 시 대안 없음 | 가상 에이전트(OC 분신)로 폴백. 또는 대기 후 재시도 |
| CX가 코드를 안 봄 | 각 모듈 완성 시 CX 코드 리뷰를 필수 게이트로 지정 |

### 4.2 프로세스 개선

| 문제 | 개선 |
|------|------|
| 승인 없이 단계 진행 | 각 Phase 산출물을 고객에게 보고하고 피드백 수령 후 다음 Phase |
| 가시성 없음 | Phase 완료 시 상태 보고서 자동 생성. GitHub에 항상 최신 상태 표시 |
| 회고가 형식적 | 회고에 "고객이 불만족할 것" 항목 필수 포함 |
| QA가 단위 테스트만 | 시각 검증, 플레이 검증, 고객 시나리오 테스트 추가 |

### 4.3 인도 개선

| 문제 | 개선 |
|------|------|
| 실행 환경 미합의 | 인도 전 지원 플랫폼 명시. 필요시 크로스 컴파일 |
| 사용자 가이드 없음 | 실행 방법 + 조작법 + 게임 흐름을 README 상단에 |
| 스크린샷 없음 | 각 화면 캡처 후 README에 삽입 |
| 릴리즈 노트 없음 | 버전, 변경사항, 알려진 이슈 문서화 |

### 4.4 태도 개선

| 문제 | 개선 |
|------|------|
| 고객에게 질문만 함 | 전문가로서 판단 제시. "A와 B가 있는데 A를 추천합니다. 이유은..." |
| 코딩에만 집중 | 기획, 설계, 문서, 테스트, 인도를 동등하게 중요하게 취급 |
| 학교 시험식 대응 | 실무자처럼 행동. 완벽한 답이 아니라 완성된 제품을 내놓기 |
| 실패를 숨기려 함 | 실패를 기록하고 분석하는 게 SDLC의 본질 |

## 5. naia-adk 이식 시 반영 사항

이 체계 실패는 하네스 아키텍처 설계에 직접 반영되어야 함:

1. **Receiver Proxy를 에이전트 풀에 필수 역할로 포함**
2. **각 Phase 종료 시 고객 보고 + 피드백 수령을 강제 게이트로**
3. **회고 템플릿에 "고객 불만족 시나리오" 필수 항목 추가**
4. **QA에 시각/경험 검증 레이어 추가 (단위 테스트만으로 불충분)**
5. **에이전트 간 코드 리뷰를 필수 게이트로** (생성자 ≠ 검증자)
